# v1.1 Current Contract Note (App + QA)

Date: 2026-04-30  
Source baseline: `reports/api-compatibility-matrix-2026-04-30-phase2-baseline.md`

## Purpose

This note defines what is considered stable enough for immediate Flutter + QA execution while the broader Target v1.1/v2 migration continues.

## Stable now (use in canary)

1. `/api/v1/*` adapter routes are active and routed through the same API wrapper as `/api/*`.
2. `x-request-id` is supported end-to-end (echoed/generated by wrapper).
3. Billing idempotency is active for:
   - `POST /api/billings` and `POST /api/v1/billings`
   - `PATCH /api/billings/:id` and `PATCH /api/v1/billings/:id`
4. Envelope mode is available via `x-bf-response-envelope: v1` (opt-in canary header).
5. Idempotency guardrail counters are live:
   - `idempotency_replay`
   - `idempotency_conflict`
   - `idempotency_wait`
   - `idempotency_expired_reclaim`
6. Idempotency metrics endpoint is available at `GET /api/ops/idempotency-metrics` (admin/superadmin).

## In progress (do not assume final contract yet)

1. GraphQL read slice is implemented and test-covered (`products`, `categories`, `widgetSettings`, `customerLookup`, `reviews`) but remains Phase 3 rollout scope.
2. Standard envelope/error contract is currently canary opt-in, not global default.
3. Some path/shape differences from draft contract still exist (`/auth/*`, nested branch config/tables/kitchens, status transition endpoint semantics).

## Not yet implemented (future phases)

1. WebSocket v1 event stream (`/ws/v1`) and event delivery semantics: Phase 4.
2. Webhook signing + retry framework: Phase 5.

## QA execution guidance

1. Test both `/api/*` and `/api/v1/*` for parity on critical flows.
2. In canary builds, always send:
   - `x-bf-response-envelope: v1`
   - `x-request-id`
   - `Idempotency-Key` on billing writes
3. Treat `409 IDEMPOTENCY_CONFLICT` as non-retriable unless key changes.
